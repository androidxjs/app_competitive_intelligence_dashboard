# App Competitive Intelligence Dashboard 产品文档

中文名：App 竞品雷达  
文档版本：v0.1 Draft  
更新时间：2026-07-06  
目标形态：面向 App 团队、独立开发者和增长团队的竞品监控与洞察工作台

## 1. 产品一句话

App 竞品雷达是一个面向 App 产品团队的 Competitive Intelligence Dashboard。用户输入自己的 App 和竞品 App 链接后，系统持续监控 App Store、Google Play、竞品官网、价格页、发布日志、截图文案、用户评论和版本变化，并自动生成竞品档案、评论洞察、功能差距、战略变化判断和每周竞品报告。

## 2. 产品定位

本产品不做泛化的“竞品分析生成器”，而是做垂直于 App 场景的持续竞品雷达。

核心差异：

| 维度 | 普通竞品分析工具 | App 竞品雷达 |
| --- | --- | --- |
| 分析对象 | 公司、官网、公开文章 | App 商店页、评论、版本、价格、截图、官网、发布日志 |
| 输出形式 | 一次性 SWOT 或市场报告 | 可追踪、可告警、可复盘的持续情报 |
| 主要价值 | 看起来完整 | 指导产品迭代、功能取舍、定价和增长 |
| 数据重点 | 宏观描述 | 用户真实评论、版本更新、功能变化、价格变化 |
| 使用频率 | 立项时偶尔使用 | 每周、每次竞品更新、每次规划版本时使用 |

## 3. 目标用户

| 用户类型 | 典型需求 | 使用场景 |
| --- | --- | --- |
| App 产品经理 | 知道竞品最近做了什么、用户在抱怨什么、下一版该做什么 | 版本规划、需求评审、老板汇报 |
| 独立开发者 | 快速掌握市场机会，不错过竞品变化 | 选题验证、功能优先级、定价策略 |
| 增长 / ASO 团队 | 分析竞品标题、关键词、截图、价格、评价趋势 | ASO 优化、落地页优化、订阅转化 |
| 创业团队负责人 | 低成本持续跟踪关键竞品和战略变化 | 周会、融资材料、产品方向判断 |
| 市场 / 竞品分析人员 | 减少手工搜集和整理成本 | 周报、竞品库、专题分析 |

## 4. 核心问题

App 团队通常面临这些问题：

1. 竞品变化分散在 App Store、Google Play、官网、博客、社媒、帮助中心和价格页中，人工跟踪成本高。
2. 竞品用户评论数量大，手动阅读效率低，难以提炼真实痛点和功能请求。
3. 版本更新日志往往很短，需要结合截图、描述、评论和官网变化才能判断真实产品方向。
4. 价格、订阅套餐、免费试用、内购结构变化不明显，容易被团队错过。
5. 产品规划时很难快速回答：竞品有什么功能我们没有？我们有什么优势？哪些功能值得跟进，哪些不值得？
6. 传统 AI 报告容易产生空泛总结，缺少证据来源、趋势跟踪和可执行建议。

## 5. 产品原则

1. App-first：优先围绕 App 商店页、评论、版本、截图、价格、ASO 信号设计能力。
2. Evidence-first：每个洞察都要能回溯到评论、版本日志、截图文案、网页变化或价格变化。
3. Continuous：不是一次性报告，而是持续监控、告警和周报。
4. Actionable：输出必须服务于产品决策，例如功能机会、需求池、风险、定价建议。
5. Human-in-the-loop：AI 给出结论、证据和置信度，最终由用户确认、收藏、忽略或转成需求。
6. Compliance-aware：优先使用公开页面、官方 API、用户授权数据和合规抓取，避免绕过登录、付费墙或平台限制。

## 6. 核心功能地图

### 6.1 App 竞品档案生成器

输入竞品 App Store / Google Play 链接，生成结构化竞品档案。

| 模块 | 内容 |
| --- | --- |
| 基础信息 | App 名称、开发者、分类、评分、评论数、版本号、更新时间、支持地区 |
| 产品定位 | 一句话定位、目标用户、核心场景、主打价值 |
| 商店页卖点 | 标题、副标题、描述、截图文案、预览视频信息 |
| 功能列表 | 从描述、截图、更新日志、评论中提取功能点 |
| 价格结构 | 免费、订阅、内购、试用、套餐、区域价格 |
| 用户反馈 | 高频好评、差评、功能请求、Bug、性能问题、付费抱怨 |
| ASO 线索 | 标题关键词、副标题关键词、描述关键词、截图卖点关键词 |
| 差异化机会 | 竞品未满足的痛点、用户反复提到但未解决的问题 |

关键输出：

- 单个竞品档案页
- 多竞品对比表
- Markdown / PDF / Excel 导出
- 可复制到周报或产品需求文档的摘要

### 6.2 App 评论洞察分析器

围绕用户评论提炼产品机会，是 MVP 的核心模块。

能力范围：

| 能力 | 说明 |
| --- | --- |
| 评论抓取 | 按 App、国家 / 地区、时间范围、评分、版本抓取评论 |
| 评论清洗 | 去重、语言识别、垃圾评论过滤、翻译、版本归因 |
| 自动聚类 | 将大量评论聚类为痛点、功能请求、Bug、价格、体验、性能等主题 |
| 情绪分析 | 判断正面、负面、中性，以及情绪强度 |
| 功能请求提取 | 提取用户明确提出的“希望增加”、“缺少”、“能不能支持”等需求 |
| Bug / 崩溃提取 | 提取闪退、卡顿、同步失败、登录异常、支付问题等反馈 |
| 付费抱怨提取 | 提取价格太贵、订阅诱导、试用取消困难、功能锁定等问题 |
| 产品机会生成 | 将评论主题转成可评估的产品建议 |
| 需求池生成 | 可将高频功能请求转入“候选需求池” |

评论洞察示例：

| 用户反馈主题 | 涉及 App | 频率 | 严重程度 | 产品机会 |
| --- | --- | ---: | --- | --- |
| 订阅价格太高 | 竞品 A、竞品 C | 高 | 中 | 设计低价轻量套餐或一次性买断选项 |
| 同步失败 | 竞品 B | 中 | 高 | 强化多设备同步稳定性，作为差异化卖点 |
| 新手不会用 | 竞品 A | 高 | 中 | 优化首次引导和模板推荐 |
| 缺少小组件 | 竞品 C | 中 | 中 | 增加桌面 Widget 作为快速跟进功能 |

### 6.3 App 版本更新监控器

持续监控竞品版本变化，并判断产品方向。

监控内容：

- 版本号变化
- 更新时间
- Release Notes 内容变化
- 标题 / 副标题 / 描述变化
- 截图变化
- 截图文案变化
- 订阅 / 内购价格变化
- 评分和评论数量变化
- 评论主题变化
- 关键词变化
- 新功能表述变化

更新洞察示例：

> 竞品 A 在 2026-07-02 更新到 v4.8.0。更新日志强调 AI 生成、模板库和团队协作；截图新增了“AI Assistant”页面；官网首页新增“团队工作流”文案。系统判断该竞品可能正在从个人工具定位转向 AI 协作工作流。

### 6.4 Competitive Research AI Agent

这是产品中的自动研究代理，负责按任务持续搜集公开情报、归档证据、生成分析并触发报告。

这里的“开源内容”建议理解为公开来源情报，也就是 public-source / open-source intelligence，而不是一定要把产品代码开源。

Agent 任务类型：

| Agent 任务 | 说明 |
| --- | --- |
| App Store Agent | 定期检查 App 商店页、版本、评分、评论、截图和价格 |
| Google Play Agent | 定期检查 Google Play 页面、评分、评论、版本和价格 |
| Website Watcher Agent | 定期抓取竞品官网、价格页、博客、帮助中心、文档页 |
| Pricing Agent | 监控订阅套餐、免费试用、折扣、区域价格、付费功能变化 |
| Launch Monitor Agent | 监控新产品发布、新功能上线、Product Hunt / 博客 / changelog 发布 |
| Strategy Signal Agent | 从文案、招聘、页面结构、功能重点变化中推断战略变化 |
| Feature Gap Agent | 对比“我的 App”和竞品功能矩阵，输出差距、优势和跟进建议 |
| Report Agent | 每周自动生成竞品周报，并发送到邮箱、Slack、飞书或 Notion |

Agent 执行流程：

1. 读取用户配置的竞品、地区、频率和重点关注主题。
2. 抓取或调用数据源，保存原始快照。
3. 对比上一次快照，识别结构化变化。
4. 使用 AI 对变化进行分类、摘要和影响判断。
5. 给每条洞察附上证据、来源、时间、置信度。
6. 根据规则触发告警或写入周报草稿。
7. 用户确认后，沉淀到竞品档案、需求池或报告库。

### 6.5 竞品网站定期抓取

除 App 商店外，竞品官网也是战略变化的重要来源。

监控页面：

- 官网首页
- Pricing / Plans 页面
- Blog / News / Changelog
- Help Center / Docs
- Feature pages
- Release notes
- Enterprise / Team 页面
- API / Developer 页面
- Terms / Subscription policy 页面

监控方式：

| 类型 | 说明 |
| --- | --- |
| 文本 diff | 检测标题、副标题、CTA、功能描述、价格文案变化 |
| 结构 diff | 检测新增页面、新增模块、新增导航项 |
| 截图 diff | 记录页面视觉变化，适合发现重点功能或品牌转向 |
| 语义 diff | AI 总结“这次变化意味着什么” |
| 证据归档 | 保存 URL、抓取时间、旧版本摘要、新版本摘要 |

### 6.6 价格变化监控

监控竞品商业化策略变化。

监控对象：

- App Store / Google Play 内购和订阅信息
- 官网价格页
- 免费试用天数
- 月付 / 年付价格
- 团队版 / 企业版价格
- 限时折扣
- 功能锁定变化
- 区域价格变化
- 退款 / 取消订阅说明变化

价格洞察输出：

| 变化 | 可能含义 | 建议 |
| --- | --- | --- |
| 年付价格下降 | 竞品可能在提升年付转化 | 观察其截图和商店文案是否同步强调折扣 |
| 新增低价套餐 | 竞品想降低付费门槛 | 评估我方是否需要轻量套餐 |
| 免费试用从 7 天改为 3 天 | 竞品可能更重视短期转化 | 监控差评中是否出现订阅抱怨 |

### 6.7 产品发布监控

监控竞品新功能、新产品线和重大版本。

信号来源：

- App Release Notes
- App 商店截图变化
- 官网新页面
- Blog / changelog
- 帮助中心新增文章
- 邮件订阅内容
- Product Hunt / Hacker News 等公开发布平台
- GitHub release 或公开 roadmap

发布事件类型：

| 类型 | 示例 |
| --- | --- |
| 新功能发布 | 新增 AI Assistant、Widget、团队空间 |
| 新产品线 | 从个人 App 扩展到团队版、Web 版、API |
| 商业化发布 | 新增 Pro、Team、Enterprise 套餐 |
| 平台扩展 | 从 iOS 扩展到 Android、Web、macOS |
| 品牌升级 | 改名、改 icon、改官网定位 |

### 6.8 战略变化监控

战略变化不是单一字段变化，而是多个信号的组合判断。

可监控的战略信号：

| 信号 | 可能含义 |
| --- | --- |
| 商店截图从个人场景变成团队场景 | 从个人用户转向团队协作 |
| 官网新增 Enterprise 页面 | 开始进军 B2B 或大客户 |
| 价格页新增 Team 套餐 | 商业模式从个人订阅扩展到团队订阅 |
| Release Notes 高频出现 AI | 将 AI 作为核心产品方向 |
| 帮助中心新增 API 文档 | 可能开放平台或生态 |
| 招聘页新增 Growth / Sales 岗位 | 进入增长或商业化扩张阶段 |
| 评论里大量抱怨广告 | 可能正在加强广告变现 |

战略变化输出应包含：

- 变化摘要
- 证据列表
- 置信度
- 对我方的影响
- 建议动作
- 是否加入周报

### 6.9 功能差距与优势分析

这是产品从“看竞品”走向“指导我方路线图”的关键功能。

输入：

- 我的 App 链接或手动录入功能列表
- 竞品 App 链接
- 目标用户和核心场景
- 关注维度，例如 AI、协作、订阅、模板、编辑能力、导出能力

输出：

| 分析项 | 说明 |
| --- | --- |
| 竞品有而我方没有 | 可作为跟进候选，但需要评估是否符合定位 |
| 我方有而竞品没有 | 可作为营销卖点和截图文案重点 |
| 双方都有但竞品更强 | 需要判断是否补齐体验差距 |
| 双方都有但我方更强 | 可作为差异化优势 |
| 用户强需求但竞品没做好 | 最值得关注的机会区 |
| 竞品做了但用户不买账 | 可避免盲目跟进 |

功能矩阵示例：

| 功能 | 我的 App | 竞品 A | 竞品 B | 用户需求强度 | 建议 |
| --- | --- | --- | --- | --- | --- |
| AI 生成 | 部分支持 | 强支持 | 不支持 | 高 | 补齐核心流程，不必复制全部高级能力 |
| Widget | 不支持 | 支持 | 支持 | 中 | 可作为低成本跟进功能 |
| 团队协作 | 不支持 | 支持 | 部分支持 | 中 | 若当前定位是个人工具，可暂不跟进 |
| 一次性买断 | 支持 | 不支持 | 不支持 | 高 | 强化为价格差异化卖点 |

### 6.10 自动周报生成

每周自动生成“竞品雷达周报”，帮助团队持续复盘。

周报结构：

1. 本周摘要
2. 重点竞品变化
3. 版本更新汇总
4. 价格与订阅变化
5. 用户评论热点变化
6. 新增功能和产品发布
7. 可能的战略变化
8. 功能差距与机会
9. 对我方的建议动作
10. 附录：证据链接、截图、原始评论样本

周报可配置：

- 每周一自动生成
- 发送到邮箱、Slack、飞书、Notion、Webhook
- 支持 Markdown / PDF / HTML
- 支持只发摘要或完整报告
- 支持人工审核后发送

## 8. 关键用户流程

### 8.1 首次创建项目

1. 用户输入自己的 App Store / Google Play 链接。
2. 用户输入 3 到 5 个竞品 App 链接。
3. 用户选择目标国家 / 地区。
4. 用户选择分析周期：7 天、30 天、90 天。
5. 用户选择重点关注：功能、Bug、付费、UI、留存、订阅、广告、AI。
6. 系统生成初始竞品档案和第一份分析报告。

### 8.2 查看评论洞察

1. 用户进入 Review Insights。
2. 选择竞品、地区、评分、版本、时间范围。
3. 系统展示痛点主题、功能请求、Bug 反馈、付费抱怨。
4. 用户点击某个主题查看原始评论证据。
5. 用户将有价值的主题加入需求池。

### 8.3 查看竞品更新

1. 系统发现竞品版本更新。
2. 自动抓取新旧版本 Release Notes、截图、描述、价格。
3. AI 总结变化并判断影响。
4. 若变化重要，系统发送告警。
5. 用户确认后加入周报。

### 8.4 做功能差距分析

1. 用户选择“我的 App”和若干竞品。
2. 系统汇总双方功能列表。
3. 用户可手动修正功能归类。
4. 系统生成 feature matrix。
5. AI 输出应跟进、可忽略、可作为优势强化的建议。

### 8.5 自动生成周报

1. 每周固定时间触发 Report Agent。
2. 系统汇总本周所有变化和洞察。
3. AI 生成周报草稿。
4. 用户审核、编辑、确认。
5. 系统导出或发送到团队渠道。

## 9. 数据源设计

| 数据源 | 数据类型 | 获取方式 | 优先级 |
| --- | --- | --- | --- |
| App Store 页面 | 基础信息、版本、截图、描述、评分、评论 | 公开页面 / 合规抓取 / 第三方 API | P0 |
| Google Play 页面 | 基础信息、版本、截图、描述、评分、评论 | 公开页面 / 合规抓取 / 第三方 API | P0 |
| App Store Connect | 自家 App 评论和数据 | 用户授权官方 API | P1 |
| Google Play Console | 自家 App 质量和评论 | 用户授权官方 API | P1 |
| 竞品官网 | 定位、功能、价格、发布内容 | 合规抓取 / RSS / sitemap | P1 |
| Blog / Changelog | 发布、路线图、战略变化 | RSS / 页面抓取 | P1 |
| Pricing 页面 | 订阅、套餐、试用、商业化变化 | 页面抓取 / 结构化解析 | P1 |
| 截图 OCR | 截图文案、卖点、功能 | OCR / 多模态 AI | P1 |
| Product Hunt / HN / GitHub Release | 新产品发布、开源项目变化 | 公开页面 / API | P2 |
| 社媒公开内容 | 宣传重点、发布节奏 | API / 合规抓取 | P2 |

## 10. AI 分析能力

### 10.1 评论分类标签

基础标签：

- Positive Feedback
- Negative Feedback
- Feature Request
- Bug / Crash
- Performance
- Pricing / Subscription
- UX / Onboarding
- Ads / Monetization
- Login / Account
- Sync / Cloud
- Export / Sharing
- Localization
- Customer Support

### 10.2 洞察对象结构

每条 AI 洞察都应保存结构化字段：

| 字段 | 说明 |
| --- | --- |
| insight_id | 洞察唯一 ID |
| title | 洞察标题 |
| summary | 一句话摘要 |
| category | 评论、版本、价格、功能、战略等 |
| competitors | 涉及竞品 |
| severity | 低 / 中 / 高 |
| confidence | 0 到 100 |
| evidence | 来源 URL、评论样本、截图、旧值、新值 |
| impact | 对我方产品的潜在影响 |
| recommendation | 建议动作 |
| status | 新发现、已确认、已忽略、已转需求 |

### 10.3 AI 输出要求

1. 不输出无证据结论。
2. 不把普通文案变化过度解读为战略变化。
3. 重大判断必须给出多个信号支撑。
4. 涉及“竞品战略”的判断要标注为推测。
5. 用户可以查看原始来源。
6. 用户可以纠正分类，系统应学习团队偏好。

## 12. 明确不做或暂缓

第一版不建议做：

1. APK / IPA 逆向分析。技术复杂且存在合规风险。
2. 无来源的 SWOT 长报告。容易空泛，不能指导行动。
3. 下载量和收入估算。没有可靠第三方数据源时准确性较低。
4. 大而全的市场研究。容易偏离 App 团队真实工作流。
5. 绕过登录、付费墙、验证码或平台限制的抓取。

## 15. 技术产品架构

建议逻辑架构：

1. Source Connectors：App Store、Google Play、网站、RSS、价格页、第三方 API。
2. Scheduler：定时任务、重试、频率控制、任务队列。
3. Snapshot Store：保存每次抓取的原始快照和结构化字段。
4. Diff Engine：对比前后版本，识别文本、结构、价格、截图变化。
5. AI Insight Engine：评论分类、功能提取、战略推断、建议生成。
6. Evidence Store：保存 URL、截图、评论样本、旧值、新值。
7. Dashboard API：向前端提供项目、竞品、洞察、报告、告警数据。
8. Report Generator：生成 Markdown、HTML、PDF、Excel。
9. Notification Service：邮件、Slack、飞书、Webhook。

## 17. 权限与合规

1. 默认只采集公开可访问数据。
2. 如果用户连接自家 App Store Connect 或 Google Play Console，必须走用户授权。
3. 抓取频率要可控，避免对目标网站造成压力。
4. 遵守 robots.txt、平台条款和 API 限制。
5. 不绕过验证码、登录态、付费墙、反爬机制。
6. 评论样本导出时注意隐私保护，避免展示不必要的用户个人信息。
7. 企业版需支持数据隔离、权限控制和审计日志。

## 18. 路线图建议

### Phase 0：原型验证，1 到 2 周

- 支持手动输入 App Store / Google Play 链接
- 抓取基础信息、版本、描述、部分评论
- 使用 AI 输出评论洞察和竞品档案
- 生成 Markdown 报告

目标：验证“评论洞察 + 版本更新”是否能让用户觉得有价值。

### Phase 1：MVP，3 到 6 周

- 项目 / 竞品管理
- 评论聚类和功能请求提取
- Release Notes 监控
- 功能矩阵
- 周报生成
- Markdown / Excel 导出
- 证据回溯

目标：形成可反复使用的 App 竞品工作流。

### Phase 2：持续监控，6 到 10 周

- 定时任务
- 价格监控
- 官网 / changelog 抓取
- 告警通知
- 截图 OCR
- 周报自动发送

目标：让产品从“分析工具”变成“竞品雷达”。

### Phase 3：团队化与 Agent，10 周以后

- Competitive Research AI Agent
- 战略变化推断
- 需求池和团队协作
- 多数据源接入
- API / Webhook
- 企业权限

目标：成为 App 团队长期使用的竞品情报系统。

## 19. 首版成功标准

第一版不以“报告有多长”为成功标准，而以“是否能指导产品行动”为标准。

建议成功标准：

1. 用户输入 3 个竞品后，10 分钟内得到一份可读的竞品简报。
2. 报告中至少有 5 条可回溯证据的用户痛点或功能机会。
3. 用户能清楚看到竞品最近版本更新了什么。
4. 用户能得到一张“我方 vs 竞品”的功能差距表。
5. 周报可以直接发给产品团队讨论。
6. 用户愿意每周回来查看更新。
